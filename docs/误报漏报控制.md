# 误报漏报控制

## 问题定义

AI PR Review 存在两类典型质量风险：

- 误报：模型指出不存在或不重要的问题，增加开发者处理成本。
- 漏报：模型没有发现真实风险，导致 Review 辅助价值下降。

本项目通过输入约束、输出结构化、长度提示和人工复核友好的报告格式来降低风险。

## 输入侧控制

1. **只基于真实 diff**

   prompt 明确要求模型不要臆造文件、函数或变更，只能基于提供的 PR 上下文分析。

2. **保留文件级信息**

   每个文件保留 `filename`、`status`、`additions`、`deletions` 和 `patch`，使模型可以结合文件路径和变更规模判断风险。

3. **截断提示透明化**

   当 patch 被截断时，系统会在上下文中写入截断说明。模型和用户都能意识到报告可能缺少部分上下文，降低因信息不完整导致的过度判断。

## 输出侧控制

1. **结构化字段**

   模型必须输出 `summary`、`riskLevel`、`risks`、`suggestions`、`testSuggestions` 和 `markdownReport`。结构化结果便于用户快速定位重点，也便于前端稳定展示。

2. **风险等级**

   `riskLevel` 帮助用户区分低风险、中风险和高风险，避免所有建议都被视为同等优先级。

3. **测试建议单独输出**

   测试建议从 Review 建议中独立出来，使用户可以更直接地补充验证场景。

4. **Markdown 报告可编辑**

   生成的 Markdown 报告可以复制到 PR 评论区，用户可以在提交前人工删改，保留最终控制权。

## 异常兜底

如果模型返回不是合法 JSON，后端不会直接中断流程，而是将原始内容放入 `markdownReport`，并返回兜底结构。这样可以减少格式波动对用户体验的影响。

## 当前限制

- 系统目前主要分析 GitHub API 返回的 diff patch，不读取完整仓库上下文。
- 对跨文件调用链、运行时行为和业务语义的判断仍需要人工复核。
- 大型 PR 中被截断的 patch 可能导致部分风险未被模型看到。

## 后续改进

- 引入项目规则配置，例如安全规则、测试规范和代码风格。
- 支持按文件分批 Review，减少大型 PR 上下文丢失。
- 增加多模型交叉评审，对高风险结论进行二次确认。
- 记录用户反馈，用于优化 prompt 和风险分类。
