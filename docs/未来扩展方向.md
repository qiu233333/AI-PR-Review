# 未来扩展方向

## GitHub App 集成

当前系统由用户手动输入 PR 链接触发分析。后续可以封装为 GitHub App，在 PR 创建、更新或评论命令触发时自动运行 Review，并将 Markdown 报告回写到 PR 评论区。

## 多模型评审

后续可支持多个模型并行或串行评审，例如 DeepSeek、OpenAI、通义千问等。系统可以比较不同模型的风险识别结果，并对高风险问题进行二次确认。

## 文件级 Review

大型 PR 的整体报告可能较长。后续可以按文件生成 Review 结果，包括：

- 文件变更摘要。
- 文件级风险等级。
- 文件级 Review 建议。
- 文件级测试建议。

最后再生成全局总结，提升可读性。

## 项目规则配置

不同团队对 Review 的关注点不同。后续可支持配置：

- 代码风格要求。
- 测试覆盖要求。
- 安全检查规则。
- 禁止修改的关键文件。
- 风险等级定义。

这些规则可加入 prompt，提升 Review 建议与项目实际规范的匹配度。

## 缓存与异步任务

后续可加入 Redis 或数据库缓存 GitHub API 响应和模型分析结果。对于大型 PR，可使用任务队列异步执行分析，前端展示任务状态，避免 HTTP 请求长时间阻塞。

## CI/CD 集成

系统可以与 CI 流程集成，在 PR 检查阶段自动运行 AI Review。对于高风险结果，可以设置为提醒或阻断策略，但最终是否阻断应由团队规则决定。

## 反馈闭环

用户可以对模型建议标记“有用”“误报”“已采纳”。这些反馈可用于优化 prompt、调整风险分类，并为后续模型评估提供数据基础。

## 更完整的上下文获取

当前主要依赖 PR diff。未来可以按需补充：

- 相关文件完整内容。
- 测试文件。
- 依赖配置。
- 最近提交历史。
- 项目 README 或规范文档。

这样可以提升模型对跨文件逻辑和业务语义的理解能力。
