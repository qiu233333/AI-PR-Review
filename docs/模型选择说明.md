# 模型选择说明

## 选择目标

本项目需要模型完成代码 diff 理解、风险识别、Review 建议生成和 Markdown 报告组织。模型选择重点关注以下能力：

- 对代码变更和 diff 格式的理解能力。
- 中文结构化表达能力。
- 稳定输出 JSON 的能力。
- 接口集成成本和替换成本。
- 响应速度与成本之间的平衡。

## 当前模型

当前默认模型为 DeepSeek `deepseek-v4-flash`，通过 OpenAI 兼容 Chat Completions 格式调用。

相关环境变量：

| 变量名 | 说明 |
| --- | --- |
| `DEEPSEEK_API_KEY` | DeepSeek API Key，必填 |
| `DEEPSEEK_BASE_URL` | DeepSeek API 地址，默认 `https://api.deepseek.com` |
| `DEEPSEEK_MODEL` | 模型名称，默认 `deepseek-v4-flash` |

## 选择 DeepSeek 的原因

1. **OpenAI 兼容接口**

   DeepSeek 支持 OpenAI 兼容调用格式，后端可以使用标准 `/chat/completions` 请求结构，降低集成复杂度，也方便后续替换其他兼容模型。

2. **适合代码评审任务**

   PR Review 需要模型理解自然语言描述、代码 diff 和结构化输出要求。`deepseek-chat` 适合通用代码理解和中文报告生成，能够满足比赛演示阶段的主要需求。

3. **便于结构化输出**

   后端通过 prompt 和 `response_format` 要求模型返回 JSON，便于前端稳定展示 `summary`、`riskLevel`、`risks`、`suggestions`、`testSuggestions` 和 `markdownReport`。

4. **便于扩展**

   模型名称通过 `DEEPSEEK_MODEL` 配置，不需要修改业务代码即可切换模型。

## 输出格式约束

后端要求模型输出以下字段：

- `summary`：PR 变更总结。
- `riskLevel`：风险等级。
- `risks`：风险代码或风险点列表。
- `suggestions`：Review 建议。
- `testSuggestions`：测试建议。
- `markdownReport`：可复制到 PR 评论区的 Markdown 报告。

如果模型返回不是合法 JSON，后端会将原始文本放入 `markdownReport`，并提供兜底字段，避免接口完全失败。

## 后续模型评估方向

后续可从以下维度评估不同模型：

- 对真实风险的召回率。
- 对无风险代码的误报率。
- 输出格式稳定性。
- 中文表达质量。
- 大型 PR 场景下的响应速度。
- 单次 Review 成本。
