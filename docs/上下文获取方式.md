# 上下文获取方式

## 目标

上下文获取的目标是将 GitHub PR 中分散的信息整理成模型可以直接分析的文本，同时避免 prompt 过长导致响应慢、失败或成本升高。

## 输入

用户输入 GitHub PR 链接：

```text
https://github.com/{owner}/{repo}/pull/{number}
```

## 链接解析

后端使用 `parsePrUrl` 完成链接解析，输出：

- `owner`
- `repo`
- `pullNumber`

解析阶段会处理以下异常：

- 空字符串。
- 非 GitHub 链接。
- GitHub 仓库链接但不是 PR 链接。
- PR number 不是数字。

## GitHub 数据获取

后端使用 GitHub REST API 获取两类数据。

PR 基础信息：

- `title`
- `body`
- `user.login`
- `html_url`

changed files：

- `filename`
- `status`
- `additions`
- `deletions`
- `changes`
- `patch`

`GITHUB_TOKEN` 为可选环境变量。未配置时可以访问公开仓库，但更容易受到 GitHub API 限流影响。

## diff 整理

`chunkDiff` 对 changed files 做标准化处理，保留模型分析所需字段：

- `filename`
- `status`
- `additions`
- `deletions`
- `patch`

对于过长 patch，系统会截断并插入提示，说明原始长度和保留长度。这样既保留主要上下文，也明确告知模型和用户存在截断。

## prompt 构建

`promptBuilder` 将以下信息组合成 prompt：

- PR 标题。
- PR 描述。
- PR 作者。
- PR 地址。
- 仓库和 PR 编号。
- changed files 列表。
- 每个文件的 patch。
- diff 长度控制统计信息。

prompt 明确要求模型输出：

- 变更总结。
- 风险等级。
- 风险代码。
- Review 建议。
- 测试建议。

## 上下文边界

系统只使用 GitHub API 返回的 PR 数据作为模型输入，不会主动读取仓库完整代码，也不会根据文件名猜测未提供的上下文。该设计能降低实现复杂度和数据权限风险，但对大型重构 PR 的深层语义理解仍有限，后续可通过按需拉取相关文件内容扩展。
