# 架构设计

## 设计目标

AI PR Review 助手的架构目标是将“PR 数据获取、上下文整理、模型分析、结果展示”拆分为清晰的模块，使每个模块职责单一、可测试、可替换，并适合按 PR 逐步迭代。

## 总体架构

```text
Vue 3 + Vite 前端
  │
  │ POST /api/review/analyze
  ▼
Node.js + Express 后端
  ├─ utils/parsePrUrl.js
  ├─ services/githubService.js
  ├─ utils/chunkDiff.js
  ├─ services/promptBuilder.js
  └─ services/aiReviewService.js
  │
  ├─ GitHub REST API
  └─ DeepSeek OpenAI 兼容 API
```

## 前端架构

前端位于 `frontend` 目录，使用 Vue 3 + Vite 实现。页面主要由以下模块组成：

- `PrInput.vue`：负责 PR 链接输入和提交。
- `LoadingPanel.vue`：负责请求过程中的 loading 状态展示。
- `ReviewReport.vue`：负责展示结构化 Review 报告和复制 Markdown 报告。
- `frontend/src/api/review.js`：封装 `/api/review/analyze` 请求。

前端通过 `VITE_API_BASE_URL` 配置后端地址。本地开发时可以不配置该变量，使用 Vite 代理访问 `/api`；部署到不同域名时可通过环境变量切换后端地址。

## 后端架构

后端位于 `backend` 目录，使用 Node.js + Express 实现。核心链路如下：

1. 接口层接收 `POST /api/review/analyze` 请求。
2. `parsePrUrl` 校验并解析 GitHub PR 链接。
3. `githubService` 调用 GitHub REST API 获取 PR 基础信息和 changed files。
4. `chunkDiff` 对 changed files 做字段筛选和 patch 长度控制。
5. `promptBuilder` 将 PR 元信息和 diff 上下文组装为模型 prompt。
6. `aiReviewService` 使用 OpenAI 兼容格式调用 DeepSeek。
7. 接口返回结构化 Review 报告。

## 数据流

```text
prUrl
  ▼
{ owner, repo, pullNumber }
  ▼
PR title/body/author + changed files + patch
  ▼
长度受控的 prompt
  ▼
DeepSeek 模型输出
  ▼
{ summary, riskLevel, risks, suggestions, testSuggestions, markdownReport }
```

## 模块边界

- URL 解析只负责输入校验和字段提取，不访问网络。
- GitHub service 只负责获取 PR 数据，不构建 prompt。
- diff 工具只负责上下文长度控制，不调用模型。
- promptBuilder 只负责拼接模型输入，不发起 AI 请求。
- aiReviewService 只负责模型调用和输出归一化。
- 前端只负责交互、展示和复制，不直接访问 GitHub 或模型 API。

## 安全设计

- `GITHUB_TOKEN` 和 `DEEPSEEK_API_KEY` 均从环境变量读取。
- `.env` 文件不提交到仓库。
- 前端不暴露 DeepSeek API Key。
- 后端对异常进行统一包装，避免向用户暴露内部堆栈。

## 可测试性

项目在 service 层保留可注入依赖，测试可以 mock GitHub API 和 DeepSeek API，不需要真实访问外部服务。当前测试覆盖 URL 解析、GitHub 请求错误处理、diff 截断、prompt 构建、AI JSON 兜底和接口编排流程。
