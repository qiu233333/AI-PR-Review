# 响应速度优化

## 性能目标

AI PR Review 的响应时间主要受 GitHub API 请求、diff 上下文长度和模型生成速度影响。本项目的优化目标是在保证报告质量的前提下，控制 prompt 长度，减少无效等待，并向用户提供明确的 loading 状态。

## 当前优化措施

### 1. changed files 分页获取

GitHub changed files 接口按页返回文件列表。后端按页拉取，避免一次请求承担过多数据，也便于后续扩展为按文件批处理。

### 2. patch 长度控制

`chunkDiff` 对 patch 做两层控制：

- 单个文件 patch 长度限制。
- 全部文件 patch 总长度限制。

超过限制时保留 patch 前部内容，并写入截断提示。这可以显著降低 prompt 过长导致的模型响应慢或失败风险。

### 3. 结构化 prompt

prompt 使用固定结构组织 PR 元信息、diff 统计和 changed files，减少模型理解输入格式的成本，也提高输出稳定性。

### 4. 前端 loading 状态

前端在请求过程中显示“正在分析 PR”，提示用户系统正在整理 diff、构建上下文并生成报告，降低等待时的不确定感。

### 5. API Token 配置

配置 `GITHUB_TOKEN` 可以提高 GitHub API 请求额度，减少匿名请求限流导致的失败或重试。

## 后续优化方向

1. **缓存 GitHub API 响应**

   对同一个 PR 的基础信息和 changed files 做短期缓存，避免重复分析时反复请求 GitHub。

2. **异步任务队列**

   对大型 PR 可将分析任务放入队列，前端轮询任务状态或通过 WebSocket 接收结果。

3. **流式输出**

   接入模型流式返回后，可以先展示变更总结，再逐步展示风险和建议，改善体感速度。

4. **按文件分批分析**

   对大型 PR 先按文件生成局部 Review，再合并总结，降低单次 prompt 压力。

5. **差异优先级排序**

   优先发送高风险文件类型或核心目录的 diff，例如认证、安全、数据库迁移和部署配置。

## 取舍说明

当前版本优先保证实现清晰和比赛演示稳定性，因此采用同步请求方式。该方式链路直观，便于说明和调试；后续在真实生产场景中，可根据 PR 规模和并发量引入异步任务与缓存。
